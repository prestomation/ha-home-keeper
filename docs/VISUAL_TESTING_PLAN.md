# Visual testing — bake screenshots into the e2e suite

**Status: proposed (separate follow-up PR, not #81).** Today screenshots are a
*PR ritual*, not part of testing: `tests/e2e/screenshots.capture.ts` is a manual
one-off that nobody re-runs and CI never touches, while the real specs
(`tests/e2e/tests/*.spec.ts`, run by `e2e.yml`) assert behavior but never look at
pixels. The companion-capture rot — broken since #72 and unnoticed because nothing
ran it — is the direct symptom. This plan makes screenshots a *byproduct of
testing*; their inclusion in PRs becomes a bonus, not the point.

The first slice already landed in #81: `tests/e2e/tests/companions.spec.ts` locks a
rendered surface that was previously unit-only, and the companions stub fixtures give
that surface deterministic, real content. This doc is the rest.

## Levels (increasing ambition — ship in order)

### Level 1 — capture runs in CI as an artifact (cheap, do first)

Fold the capture into `e2e.yml` so every run regenerates the PNGs and uploads them as
a build artifact (alongside the existing Playwright-report upload). Removes the "did
someone re-run capture?" doubt entirely; the PR-embed becomes a copy step, not a
manual gate. Does **not** assert anything yet — the images are simply always real.

- Add a capture step to `e2e.yml` (after the spec run) invoking the existing
  `screenshots.capture.ts` against the already-running container.
- Upload `docs/images/*.png` as an artifact; optionally diff against the committed
  copies and warn (not fail) so drift is visible without blocking.

### Level 2 — visual regression on a stabilized handful (the real thing)

Use Playwright's `expect(page).toHaveScreenshot('settings-profiles.png')` against
committed baselines. CI fails on unexpected visual change; intended changes are
accepted by re-running with `--update-snapshots`.

**Hard prerequisite this codebase doesn't meet yet: determinism.** We have already
observed every capture PNG changing run-to-run purely from relative dates
("due 5/20/2026" drifts with `now`). Visual diffs are pure noise until that's fixed:

- **Pin the clock.** `recurrence.py` already takes an explicit `now`; the e2e seed
  (`home_keeper` storage doc + any sensor states) should use **absolute** dates
  against a frozen reference so "overdue / due soon" buckets are stable. Decide
  whether to freeze HA's clock or just seed dates far enough from "now" that bucket
  boundaries never move within a run.
- **Kill animation/caret/cursor.** Inject a stylesheet disabling CSS transitions and
  the text caret before each shot (Playwright supports this via `toHaveScreenshot`'s
  `animations: 'disabled'`, plus a small reset for HA's own transitions).
- **Pin viewport + device scale factor**, already mostly handled by the Playwright
  project config — make it explicit.
- **Baselines come only from the CI container.** Font hinting differs across OSes, so
  a laptop-generated baseline = perpetual false diffs. Wire `--update-snapshots` into
  `ci/e2e-up.sh` (e.g. `UPDATE_SNAPSHOTS=1 bash ci/e2e-up.sh`) so regenerating is a
  one-command, in-container operation, and document that baselines are never shot
  locally.

**Scope it.** Visual regression has a real maintenance tax. Start with 3–5
high-value, stabilized surfaces (Settings, a populated card, the task list,
Companions) — **not** all ~25 current captures. Grow only as surfaces prove stable.

### Level 3 — the gate automates itself

Once Level 2 holds, AGENTS.md's "every UI PR must embed current screenshots" softens
from human-enforced to CI-proven: a job regenerates, diffs, and updates a gallery /
PR comment. Update AGENTS.md + `.amazonq/rules/` in that same change.

## Two homes, one harness

Keep the **doc images** (`docs/images/`, curated, populated demo data, full-page,
embedded in README/PR) separate from the **regression baselines**
(`tests/e2e/*-snapshots/`, deterministic, a stable subset). They want different
things — a doc shot is *pretty*; a baseline is *stable* — but both are generated by
the same Playwright harness driving the same page helpers, so they can't drift in how
they reach a surface. The capture script and the visual specs should share the
navigation helpers in `tests/e2e/tests/helpers.ts`.

## Open questions

- Freeze HA's wall clock vs. seed absolute dates with wide margins — which is less
  fragile across HA upgrades?
- Do we diff doc images too (Level 1 warn-only), or only the dedicated baselines?
- Threshold/anti-aliasing tolerance for `toHaveScreenshot` (`maxDiffPixelRatio`) — set
  per-surface or global?
