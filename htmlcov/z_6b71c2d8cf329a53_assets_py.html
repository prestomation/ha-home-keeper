1"""Asset (appliance) model helpers for Home Keeper. 

2 

3An *asset* is a physical thing you keep — a fridge, a furnace, an under-sink RO 

4unit — that maintenance tasks (and, later, batteries / other metadata) hang off 

5of. Like tasks, an asset is stored as a plain JSON-serializable ``dict`` so it 

6round-trips through HA's ``Store`` and the websocket/service APIs untouched. 

7 

8Concerns living in one record (see ``IDEAS.md`` / ``docs/DESIGN.md``): 

9 

101. **Virtual-device provision** — when an appliance has no Home Assistant device 

11 to attach to, Home Keeper registers a real registry device for it 

12 (``kind == "virtual"``). Its registry identifier is 

13 ``(DOMAIN, f"{ASSET_IDENTIFIER_PREFIX}_{asset_id}")`` so it never collides with 

14 the per-task self-owned devices, which key on the bare task id. 

152. **Asset metadata** — descriptive/temporal attributes (purchase/install/warranty 

16 dates, cost, vendor, manual link, notes) that can also enrich an *existing* 

17 device from another integration (``kind == "existing"``), without owning it. 

183. **Parts / wear items** — a structured list of components (filters, bulbs, anode 

19 rods, shade material). A ``wear`` part with a replacement interval drives a 

20 maintenance task elsewhere (see ``store.reconcile_part_tasks``). 

214. **Relationships** — ``parent_asset_id`` makes a virtual asset a *subdevice* of 

22 another virtual asset (native ``via_device`` hierarchy); ``related_device_ids`` 

23 loosely associates arbitrary registry devices (including foreign ones HA won't 

24 let us reparent), surfaced only in the panel. 

25 

26This module imports nothing from Home Assistant so it stays unit-testable; HA 

27specifics (the current time, the area/device registries) are injected or validated 

28by the caller. 

29"""

66# Free-form metadata: an ordered list of typed entries the user can shape however 

67# they like (serial numbers, warranty/purchase/install dates, provider, links…), 

68# replacing the old prescriptive per-field set. Each entry is 

69# ``{id, type, label, value[, track]}``. A ``date`` entry with ``track`` set becomes 

70# a real ``date`` sensor on the asset's device page (so e.g. "warranty expiring in 

71# 30 days -> notify" still works natively, but only for dates you opt in to track). 

72METADATA_TYPES = ("text", "link", "date") 

73 

74# Per-asset documents: an ordered list of manuals / warranties / receipts. Each entry 

75# is EITHER an external ``link`` (an http(s) ``url``) OR an uploaded ``file`` stored on 

76# disk and served back through the document HTTP view. For a ``file`` entry the binary 

77# lives under the config dir (see ``manuals.py``); the record only carries the metadata 

78# (``filename``/``content_type``/``size``) needed to locate and serve it. Each entry is 

79# ``{id, kind, name, created[, url | filename, content_type, size]}``.

278def _merge_documents(existing: list[dict], incoming: list[dict]) -> list[dict]: 

279 """Reconcile a generic asset write's ``documents`` against the stored list. 

280 

281 ``file`` documents own an on-disk blob, so they are **upload-only**: managed solely 

282 by the upload view and the ``*_asset_document`` services, never by the generic 

283 ``add_asset`` / ``update_asset`` write. This keeps the two in sync — a generic 

284 write can't inject a phantom ``file`` entry (no blob behind it) and, by always 

285 carrying the stored ``file`` entries through, can't orphan a blob by omitting one. 

286 The generic write therefore controls only the ``link`` documents. 

287 """ 

288 files = [d for d in existing if d.get("kind") == "file"] 

289 links = [d for d in incoming if d.get("kind") == "link"] 

290 return [*files, *links]

483def _merge_parts(existing: list[dict], incoming: list[dict]) -> list[dict]: 

484 """Carry the stored ``last_replaced`` and attached-file fields across an edit. 

485 

486 The panel can seed ``last_replaced`` when adding a wear item (so the derived 

487 maintenance task starts from the real date), but it is also stamped automatically 

488 on completion. To avoid a parts round-trip that doesn't re-send it clobbering that 

489 completion history, for parts that already exist (matched by ``id``) we keep the 

490 stored value unless the caller explicitly set one. ``stock``/``reorder_at`` are 

491 ordinary user-editable fields — incoming wins, including a ``None`` that clears 

492 them — so they are intentionally *not* preserved here (otherwise stock tracking 

493 could never be switched back off). ``create_buy_task``/``restock_quantity`` are 

494 likewise ordinary incoming-wins fields (so auto-buy can be toggled back off). 

495 

496 ``file_name``/``file_content_type``/``file_size`` are **always** restored from the 

497 stored part, unconditionally — like an asset's ``file`` documents, a part's 

498 attached file is upload-only (see :func:`set_part_file`) and must never be

500 ``_normalize_part`` never actually produces) are ignored outright. 

501 """

505 prior = by_id.get(part["id"]) 

506 if prior and part.get("last_replaced") is None: 506 ↛ 508line 506 didn't jump to line 508 because the condition on line 506 was always true

606# ``adjust_part_stock``. Each maps to one bus event (or none); see store.py. 

607STOCK_NONE = "none" 

608STOCK_LOW = "low" 

609STOCK_OUT = "out" 

610STOCK_RESTOCKED = "restocked"

613def stock_transition(old: int, new: int, reorder_at: int | None) -> str: 

614 """Classify a stock change from *old* to *new* as a single edge transition. 

615 

616 Returns one of ``"none" | "low" | "out" | "restocked"``. This is the pure core 

617 behind the edge-triggered stock events: it fires once *per crossing* rather than on 

618 every step. Precedence matters — ``"out"`` (reaching zero) wins over ``"low"`` so a 

619 single decrement that drops an already-low part to zero is reported as out-of-stock, 

620 not a (repeat) low-stock. A part with no ``reorder_at`` threshold is untracked and 

621 never transitions. 

622 """ 

623 if reorder_at is None: 

624 return STOCK_NONE 

625 if new == 0 and old > 0: 

626 return STOCK_OUT 

627 if new <= reorder_at and old > reorder_at: 

628 return STOCK_LOW 

629 if new > reorder_at and old <= reorder_at: 

630 return STOCK_RESTOCKED 

631 return STOCK_NONE

634def consume_part_stock(part: dict) -> str: 

635 """Decrement a part's on-hand ``stock`` by one (never below zero). 

636 

637 A no-op (``STOCK_NONE``) for parts that don't track stock. Otherwise returns the 

638 edge transition (``stock_transition``) this consumption caused, so the caller emits 

639 at most one stock event per crossing rather than on every step while already low. 

640 """ 

641 stock = part.get("stock") 

642 if stock is None: 

643 return STOCK_NONE 

644 old = int(stock) 

645 new = max(0, old - 1) 

646 part["stock"] = new 

647 return stock_transition(old, new, part.get("reorder_at"))

650def adjust_part_stock(part: dict, delta: int) -> str:
